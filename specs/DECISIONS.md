# Decisions Log

Append-only log of non-trivial decisions. Each entry: `- **D-N:** Decided X because Y (YYYY-MM-DD). [intent: I-N]`

- **D-1:** Reimplement agentskills.io conformance natively in Rust instead of shelling out to skills-ref, because P-2 forbids Python and offline use needs no external validator (2026-05-29). [intent: I-1]
- **D-2:** Validate the `name == parent_dir` rule against the install destination (the frontmatter `name`) at install time, because that is the directory actually created (2026-05-29). [intent: I-1]
- **D-3:** Abstract install behind a `SkillSource` trait with a local-dir source now, deferring live GitHub fetch, because it keeps install tests hermetic and offline (2026-05-29). [intent: I-1]
- **D-4:** Generate TypeScript IPC types from Rust via tauri-specta behind an optional `specta` feature, because one source of truth prevents drift without burdening core builds (2026-05-29). [intent: I-1]
- **D-5:** Split the backend into a Tauri-free `core` crate plus a `src-tauri` shell, because the EARS-pinned logic must build and test without the webview toolchain (2026-05-29). [intent: I-1]
- **D-6:** Adopt bun as the JS toolchain (per P-9) and delete `package-lock.json` in favor of `bun.lock`, rewiring `tauri.conf.json` before-commands and docs from `npm`/`pnpm`/`npx` to `bun`/`bunx`, because a single package/script/test runner removes npm-lockfile drift; verified `bun install`, `bun run lint`, and `bun run build` all pass (2026-05-29). [intent: I-1]
- **D-7:** Group usage analytics by the `cwd` recorded on each transcript line rather than by decoding the `~/.claude/projects/<folder>` name, because the folder encoding (slash→hyphen) is lossy and ambiguous while `cwd` is the authoritative working directory (2026-05-29). [intent: I-2]
- **D-8:** Parse transcripts tolerantly as generic `serde_json::Value`, skipping any line that fails to parse or isn't a Skill-bearing assistant line, because transcripts are an external, evolving format and read-only analytics must never crash on a malformed line (2026-05-29). [intent: I-2]
- **D-9:** Make install reachable end-to-end via "Install from folder…" in the Library — native folder picker (`tauri-plugin-dialog`, `dialog:allow-open` scoped to the main window per P-10) → typed `install_local_skill` command → the tested core engine — while the shop's GitHub-fetch install stays deferred (D-3), because the install/validate/atomic-copy engine was already built and tested but had no UI surface, leaving the app unable to install anything. The dialog only returns a chosen path; the privileged FS work stays in the Rust core (P-14) (2026-05-30). [intent: I-1]
- **D-10:** Craft is a structured editor — frontmatter fields (`name`, `description`, `license`, `compatibility`) plus a body textarea — and the frontend assembles `SKILL.md` via the `yaml` serializer, while the gate's verdict comes only from the backend `check_conformance`/`publish_skill` path. The UI renders `Finding`s but implements no rules, so the publish gate and the install gate share one validation source of truth and cannot diverge (2026-05-29). [intent: I-3]
- **D-11:** Craft publishes by writing `SKILL.md` atomically into the active target via a new single-file `atomic_write_file` (temp-file-then-rename within the dir), overwriting only `SKILL.md` and preserving sibling resource files; the new `publish_skill` core fn validates-before-write, mirroring `install`'s order (P-6). Because the form uses one `name` for both frontmatter and directory, `name == parent_dir` (D-2) holds by construction, and a malformed `name` fails validation before it is ever joined into a path. Scope covers new skills and editing an existing installed skill, loaded via `read_skill_md_at` (2026-05-29). [intent: I-3]
- **D-12:** Adopt vitest (jsdom + Testing Library, run via bun per P-9) as the frontend test runner, excluding `*.test.tsx` from the `tsc -b` production build, because the I-3 EARS outcome is pinned to a vitest test and the repo had no JS test runner; the production build stays free of testing-library types (2026-05-29). [intent: I-3]
- **D-13:** Wire the Tauri MCP bridge (`tauri-plugin-mcp-bridge`) for manual GUI verification in a production-clean way: an optional dep behind a `mcp-bridge` cargo feature (absent from release builds), with `withGlobalTauri` and the `mcp-bridge:default` capability supplied only by a dev overlay `tauri.dev.conf.json` merged via `--config` in the `tauri:dev` script. Chosen over editing `tauri.conf.json`/`capabilities/default.json` directly so the production capability set and webview exposure stay untouched (P-10, P-11); runtime `add_capability` was rejected because it requires Tauri's `unstable` feature. Verified end-to-end: bridge on `0.0.0.0:9223`, driver connected, Craft gate + publish + Library round-trip driven through the webview (2026-05-30). [intent: I-3]
- **D-14:** Implement the shop's GitHub-fetch install as a `SkillFetcher` trait in the core (the network seam promised by D-3) plus a `RemoteSkill` source that runs the *same* `install` engine as the local-folder path, exposed via a new `install_registry_skill` IPC command; the live `HttpSkillFetcher` (GitHub Contents API, recursive, shell-only) downloads an entry's files into a temp dir which `install` then validates-then-atomically-copies. Chosen so the core stays network-free and the EARS outcome is pinned to a hermetic test (`install_from_registry_fetches_validates_installs`) that injects a fake fetcher, and so a shop install can never diverge from the folder install's validate-before-write gate (P-6). Fetched file names are validated as single path components before any write (defense in depth ahead of the install-root confinement, P-12/P-14); the GitHub API base is injectable and access is unauthenticated/read-only (no secrets). The full Tauri shell cannot be compiled in the web sandbox (missing GTK/WebKit system libs), so the shell wiring is verified by inspection while the new core test, frontend lint/typecheck/build, and a new `Shop.test.tsx` install test all pass (2026-05-30). [intent: I-1]
